# P1.4 — Setup Wizard + Login + Settings shell (FRONTEND)

> Track: **FRONTEND** (Svelte 5 + Vite SPA, embedded into and served by every node).
> Scope: **Setup Wizard** (09 §1), **Login** (09 §2), **Settings shell** (09 §8), plus
> the **SPA shell**: boot/first-run probe, router with the uninitialized/auth gates,
> the auth guard, the global header, and the shared cross-cutting libs (API client,
> error-envelope handling, `If-Match` plumbing, session store) that the rest of the UI
> pieces build on.
> Consumes upstream pieces **P0.4** (web server / `internal/web` `Deps` seam + embedded
> asset serving + the HTTP API surface this UI calls) and **P1.3** (auth: admin
> password / sessions / API keys — the server side of Login + Settings).

## 1. Purpose & scope

Build the Svelte+Vite single-page-app **shell** and the three auth/admin screens that
gate and bracket the whole UI: the **Setup Wizard** (first-run, uninitialized node:
*create cluster* vs *wait to be adopted*, showing node id / CSR fingerprint / adoption
PIN — 09 §1), the **Login** screen (admin-password → session cookie — 09 §2), and the
**Settings** screen (change admin password, list/create/revoke API keys, cluster info +
CA fingerprint, danger-zone leave/reset — 09 §8). This piece owns the app boot state
machine (probe → wizard / login / app), the SPA router with the **uninitialized** and
**authenticated** route guards (09 §0 shared-state machine), the global nav shell +
header, and the shared front-end infrastructure (typed API client over `/api/v1`,
error-envelope + `409`/`If-Match` handling, the session/config-version store) that the
sibling FRONTEND pieces (Dashboard/Cluster/Groups/Node/Media) plug into. It is a
**pure frontend** piece: it emits assets to `internal/web/dist` (embedded + served by
P0.4); it adds **no Go code** beyond confirming the embed path.

## 5. Behavior & algorithms (cite docs; quote A.12 verbatim)

### 5.1 Boot / first-run detection (09 §1 "First-run detection", 08 A.1/G.1)
On every app load, probe node status. If `initialized == false` the router **forces
`/setup` and blocks all other routes** (the **uninitialized** shared state, 09 §0). If
`true`, `/setup` redirects to `/login` (or `/` if a session exists). The probe must be
**resilient**: if the probe itself errors, show the **error** state with **Retry** rather
than guessing (09 §1). (This corrects the media `App.svelte` behavior, which falls back
to "enter app" on probe failure — Ensemble must show an error instead.)

### 5.2 Setup Wizard (09 §1)
- **Two and only two** paths (D9, 09 §1): **(a) Create new cluster** and **(b) Wait to be
  adopted**. A segmented/radio selector switches the lower panel.
- **(a) Create:** `clusterName` (required), `adminPassword` + confirm with a **client-side
  strength meter**. Submit → `POST /api/v1/setup {clusterName, adminPassword, nodeName?}`
  (B.1). On success the node initializes, a **session cookie is set** (B.1 logs the operator
  in immediately) and the SPA lands on **Dashboard** (`/`). Server hashing is **argon2id**
  (09 §1 components; A.11 pins `golang.org/x/crypto/argon2`) — the meter is advisory only.
- **(b) Wait to be adopted:** read-only **Node ID**, **Fingerprint** (the node's **CSR
  public-key fingerprint** the CA will sign — 03 / 09 §1), and the **Adoption PIN** shown
  as `0 0 0 0` with a prominent note: the placeholder is `0000` but is **treated as a real
  secret** (D9). **Copy** buttons on ID + fingerprint. No write happens here — the node
  keeps advertising on discovery until adopted from another node's Cluster screen (09 §1b /
  screen 4). When adoption completes the node flips to `initialized` (learned via gossip /
  CSR signed) and the wizard **auto-advances to Login** (09 §1 user flow 2).
- **States** (09 §1): `loading` skeleton on both panels; `error` surfaces the envelope
  `code`/`message` (e.g. weak password, or already-initialized → redirect to Login); a
  hard "cannot reach this player" error if the local API is unreachable.

### 5.3 Login (09 §2)
- Single-admin model, **no username** (D11). Password field + "keep me signed in" (session
  duration) + submit. Footer chips: **cluster name**, **serving-node id**, **CA-trust**
  indicator.
- Submit → `POST /api/v1/auth/login {password}` (B.2) → `Set-Cookie:
  ensemble_session=...; HttpOnly; Secure; SameSite=Strict` → land on the route originally
  requested (default **Dashboard**). Session expiry anywhere returns the user here.
- **error:** bad password → generic "wrong password" (no user enumeration, 09 §2 / 03). If
  the node throttles, show the rate-limit message — the **adoption/auth brute-force guard**
  numbers are pinned in **A.12** verbatim: *"soft backoff after 3 consecutive fails; hard
  15-min lockout after 10 fails per 5 min; nonce TTL 30 s."* (Login surfaces the
  `code:"rate_limited"` envelope, 08 B.2.)
- **uninitialized:** redirect to Setup Wizard. **offline:** "cannot reach this player; try
  another node's URL" (any node hosts the UI).

### 5.4 Settings (09 §8) — all writes carry `If-Match` (08 §0.5)
- **Change admin password:** current + new + confirm → `POST /api/v1/auth/password
  {currentPassword,newPassword}` with **`If-Match: <configVersion>`** (B.3a; `409
  version_conflict` → reload + reapply; wrong current → field error). Updates the argon2id
  hash in `ConfigDoc.Auth` (D11).
- **API keys:** `GET /api/v1/auth/keys` (B.5, list metadata only — name/created/last-used)
  → table with **Revoke** (`DELETE /api/v1/auth/keys/{id}`, B.7, `If-Match`). **Create**
  (`POST /api/v1/auth/keys {label}`, B.6, `If-Match`) returns the **plaintext secret
  exactly once** — render it in a CopyField with a "copy it now, it won't be shown again"
  note; only hashes are stored (`Auth.apiKeyHashes`, §6.5).
- **Cluster info:** `GET /api/v1/cluster/info` (C.1) → name, created, node counts
  (online/total from `counts`), and the **CA fingerprint** (`cluster.caFingerprint`) with a
  copy button — same trust anchor shown on the Cluster screen header (09 §4).
- **Danger zone — Leave / reset cluster:** behind a **typed confirm** (ConfirmModal type-to-
  confirm) → `POST /api/v1/cluster/leave` (C.6). Coordinated self-forget: the node asks the
  cluster to add its cert fingerprint to the grow-only `RevokedSet` + drop its `NodeRecord`
  (gossiped, **monotonic union** merge — README §6.5 / A.6), then wipes local
  certs/identity/config and reboots into the **Setup Wizard**. If unreachable it wipes
  locally anyway (`coordinated:false`) — warn the operator if peers are offline (09 §8
  offline state). After completion the SPA re-probes status and lands on `/setup`.
- **States** (09 §8): section skeletons (`loading`); empty API-key list → "No API keys —
  create one for programmatic access"; `409` on any write → reload + reapply; cluster-wide
  info still renders from gossiped ConfigDoc when offline.

### 5.5 Cross-cutting: error envelope, `If-Match`, proxy/offline (09 §0, 08 §0.4–0.6)
- All non-2xx parse the envelope `{"error":{"code","message"}}` and render an inline
  **Banner** with the `code`+`message`, a **Retry**, and for `409 version_conflict` a
  **Reload & reapply** affordance (09 §0). `412 precondition_required` is treated as a bug
  guard (the client must always send `If-Match` on mutating calls).
- Read-only GETs capture the `ETag` into `configVersion` so the next write can seed
  `If-Match` (08 §0.5). `B.4`'s `configVersion` seeds it at session refresh.

> No A.12 audio tunables apply to this piece. The only A.12 numbers it surfaces are the
> **adoption/auth guard** values (§5.3) and the PIN placeholder `0000` (D9 / A.9), all
> quoted verbatim above.

## 6. Import constraints

- **Frontend only.** The UI talks to the node **exclusively over HTTP `/api/v1`** (and
  `/bootstrap/info` pre-init); it has no Go imports and therefore is structurally immune to
  the `internal/web`→engine layering rule. The relevant invariant it **upholds** is the
  server-side one (doc 01 §2): `internal/web` reaches the engine only through the
  `web.Deps` function-value seam and must **not** import `group`, `stream/*`, or `audio/*`
  — this piece must not introduce any UI endpoint that would force the web layer to import
  those packages. Everything this UI needs is already exposed by the existing `/api/v1`
  surface (08) behind `Deps` (A.14.3); no new server seam is required.
- The build output goes to `internal/web/dist` (embedded by P0.4); this piece must not
  change the embed mechanism, only populate the directory via `vite build`.

## 9. Risks / open questions

1. **First-run probe endpoint mismatch (blocking-ish).** 09 §1 cites
   `GET /api/v1/status` returning `{initialized, nodeId, fingerprint, clusterName?}`, but
   08 G.1 `GET /api/v1/status` is **auth-gated runtime telemetry** (different shape, no
   `initialized`/`fingerprint`), while the **unauthenticated** pre-init identity actually
   lives at `GET /bootstrap/info` (A.1, returns `state`/`fingerprint`/`nodeId`). **Resolution
   the orchestrator must confirm:** the wizard should probe **`/bootstrap/info`** while
   uninitialized (it is the only surface reachable before a cert/session exists) and treat
   `state=="uninitialized"|"foreign"` as not-initialized; once initialized, derive auth/
   redirect from `GET /api/v1/auth/session` (B.4). P0.4/P1.3 must expose a probe that works
   **before** any session/cert exists — confirm which path the UI calls.
2. **Login footer "CA-trust ✔" indicator (09 §2)** — there is no pre-auth endpoint that
   returns the CA fingerprint (`cluster/info` C.1 is auth-gated). The indicator can only be
   populated post-login or from `/bootstrap/info` if it carries it; for MVP it may be a
   static "served over TLS" chip until login. Confirm with P1.3.
2. **"keep me signed in" semantics** — 08 B.2 returns `expiresAt` but does not define a
   long-lived vs short session toggle. Assumed: the checkbox sets a longer server-side TTL;
   needs a body field (`keep:true`?) that P1.3 must accept, else the checkbox is cosmetic.
3. **Session vs config version for `If-Match`** — `auth/password` (B.3a) uses `If-Match`
   on the **config version**; the UI seeds it from `auth/session.configVersion` (B.4). If
   another node bumps the version between session-load and submit, expect `409` (handled),
   but rapid key churn could thrash — acceptable per 09 §0.
4. **No live-status (WS/SSE) in this piece** — 09 §0 mentions an optional live stream;
   this shell does **not** open it (that is a Dashboard/Groups sibling-piece concern). The
   shell only exposes the route slots; confirm the live store ownership boundary with the
   sibling pieces to avoid two implementations of `ws.ts`.
5. **Vitest/Playwright not in media's `package.json`** — adding test tooling is net-new
   dev-dependency surface; keep it `devDependencies` only so the embedded `vite build`
   (D24 pure-Go single binary) is unaffected.
