## 1. Purpose & scope

Build the two operator-facing Svelte screens that compose and run **Groups**: the
**Dashboard** (screen 3 — landing page: one card per group with play state, master, the
negotiated profile/transport summary, a members table with channel role + LIVE sync
error + per-node render state, sink-less "master (no local audio)" / "⊘ no sink" badges,
and an offline-nodes strip) and the **Groups** screen (screen 5 — list/create groups,
move members between groups, view/override the auto-negotiated profile codec/FEC/rate +
transport, select media, and play/stop). This is **frontend only**: every datum is read
from the P0.4 live WS `StateSnapshot` (static ConfigDoc projection) joined with the P2.7
per-group `GET /api/v1/groups/{id}/status` live telemetry; every mutation goes through the
P0.4 `lib/api.ts` REST client against the canonical `/api/v1` endpoints from
[08](../08-http-api-reference.md) with `If-Match` optimistic concurrency. No Go engine
code, no new endpoints. The screen behavior (states, flows, badges) is authoritative in
[09 §3/§5](../09-ui-screens.md); the wire shapes are authoritative in [08](../08-http-api-reference.md).

## 4. Public API (the component + store contracts this piece exposes)

TypeScript types mirror the **P0.2 web-owned view types** (the WS `StateSnapshot` payload)
and the **08 wire shapes** verbatim — do not invent alternative field names. Codec/FEC/
transport are **string enums** in JSON (README §6.5): `"pcm"|"opus"`,
`"none"|"xorParity"|"duplicate"`, `"udp"|"tcp"`.

```ts
// lib/groups.ts — types mirror P0.2 ConfigView/NodeView/GroupView (StateSnapshot.config).
export interface Profile {
  codec: 'pcm' | 'opus'
  fec: 'none' | 'xorParity' | 'duplicate'
  rate: number              // canonical Hz, 48000
  framesPerChunk: number    // 480
  fecK: number              // 8
  interleave: number        // 4
}
export interface Media { file: string; loop: boolean }
export interface Capabilities {
  render: boolean
  sinks: string[]
  encode: string[]          // EncodeCodecs (origin-side)
  decode: string[]          // DecodeCodecs (listener-side)
  fec: string[]
  maxRate: number
}
export interface NodeView {
  id: string; name: string; addrs: string[]
  hwDelayUs: number; channel: 'stereo' | 'left' | 'right'; gainDb: number
  caps: Capabilities
}
export interface GroupView {
  id: string; name: string; memberNodeIds: string[]
  profile: Profile; media: Media; playing: boolean
}

// derived stores over the live WS snapshot (P0.4 `state`)
export const groups: Readable<GroupView[]>
export const nodeById: Readable<Map<string, NodeView>>
export const groupById: Readable<Map<string, GroupView>>
export const unassignedNodeIds: Readable<string[]>   // members in no explicit group (09 §5)
export const configVersion: Readable<number>          // seeds If-Match for writes

// lib/groupStatus.ts — live telemetry from GET /api/v1/groups/{id}/status (08 §G.2)
export interface MemberStatus {
  nodeId: string
  syncErrorUs: number       // content-domain sync error (06); master => 0/omitted
  offsetUs: number
  driftRatio: number
  underruns: number
  clockQuality: 'good' | 'fair' | 'poor'
  online: boolean
}
export interface GroupStatus {
  groupId: string
  masterNodeId: string
  profile: Profile          // negotiated *result* (vs stored override)
  streamGen: number
  playing: boolean
  members: MemberStatus[]
}
// pollGroupStatus subscribes a group id to ~1 Hz polling; returns a disposer.
export function pollGroupStatus(groupId: string): () => void
export const groupStatus: Readable<Map<string, GroupStatus>>   // by group id

// lib/groupActions.ts — all require If-Match: configVersion; all may proxy (08 §0.6).
// Each resolves to the new ConfigDoc version (from ETag) or throws ApiError{code,message}.
export function createGroup(name: string): Promise<number>                         // 08 §E.2
export function patchGroup(id: string, patch: GroupPatch): Promise<number>         // 08 §E.4
export function playGroup(id: string, sel?: Media): Promise<number>                // 08 §F.3
export function stopGroup(id: string): Promise<number>                             // 08 §F.4
export function selectMedia(id: string, file: string, loop: boolean): Promise<number> // 08 §F.2
export function moveNode(nodeId: string, toGroupId: string): Promise<number>       // E.4 on both groups, one write
export interface GroupPatch {
  name?: string
  memberNodeIds?: string[]
  profile?: Partial<Profile>     // codec/fec/rate/framesPerChunk/fecK/interleave overrides
  playing?: boolean
}

// syncfmt.ts
export function syncMs(us: number): string                  // 380 -> "+0.38 ms", master -> "—"
export type SyncLevel = 'ok' | 'warn'
export function syncLevel(us: number): SyncLevel            // |us| over threshold => 'warn'
```

## 5. Behavior & algorithms (cite docs; A.12 numbers verbatim)

**Data composition (09 §3 "Data sources", §5 "Data sources").** Static structure (groups,
members, channel role, selected media, profile, transport, the stored `playing` bool) comes
from the **live WS `StateSnapshot.config`** (`ConfigView`, P0.2 §4) — equivalent to
`GET /api/v1/cluster/info` + `GET /api/v1/nodes` + `GET /api/v1/groups` composed, but pushed
at the P0.2 hub cadence so the screens are reactive without polling the config. **LIVE**
per-member sync error / master / codec-FEC-rate-in-effect / buffer health comes from
`GET /api/v1/groups/{id}/status` (08 §G.2), **polled ~1 Hz per visible group** (or consumed
from the WS stream if P2.7 multiplexes it — `groupStatus.ts` prefers WS, falls back to poll).

**Member-row join (Dashboard, 09 §3).** For each `group.memberNodeIds`, join the
`NodeView` (channel role, name, `caps.render`) with the `MemberStatus` (sync error, online):
- **Master row:** sync error shows `"—"` (it is the reference, 04 §4.1.3); render-state dot
  `● rendering` if `caps.render`, else the sink-less branch below.
- **Sink-less master (`caps.render === false`):** member row + card header carry the
  **"⊘ master (no local audio)"** badge (`MasterBadge` with the master node) — it originates
  the stream + serves the clock but renders nothing (D17, 04 §4.2). No channel role, no
  per-listener sync error.
- **Sink-less non-master member (`caps.render === false`):** **"⊘ no sink"** state, no
  channel role, no sync error (it is never a listener — 04 §4.2.4). This is a *configuration*,
  rendered normal-weight — visually distinct from offline (dimmed + offline chip).
- **Offline member (`status.online === false`, or proxied `/status` hop failed):** row
  dimmed, sync error `"—"`, `OfflineChip` "last known" (value from gossiped ConfigDoc). A
  whole-group offline (master unreachable) disables Play/Stop and shows last-known from
  gossip (09 §3 states).

**Sync-error threshold (⚠).** `syncErrorUs` is the **content-domain sync error** owned by
[06](../06-audio-output-scheduling.md). The drift loop's gross-error bound is the single
source of truth for "too far off": **A.12 — `HardErrSamp = 2400 samples (50 ms)`** at the
canonical **`rate = 48000` Hz** (A.12 Audio canonical). This piece does **not** invent a
threshold; it flags `⚠` when `|syncErrorUs|` approaches the drift loop's operating band. Per
A.12 the steady-state target is **sub-ms** (A.13 P4: "measured inter-node offset stays
sub-ms over 10 min"), so `syncLevel` warns at **≥ 1 ms** (1000 µs) — well under the
`50 ms / HardErrSamp` reseek bound but above the sub-ms acceptance target. The exact warn
edge is surfaced as a single constant `SYNC_WARN_US = 1000` with a doc-comment citing A.12
HardErrSamp + A.13 P4 (see §9 open question — confirm the warn edge with the 06 author).

**Profile block (09 §5).** Render `profile.codec`/`fec`/`rate` as the **auto-negotiated
value** (from `GroupStatus.profile`, the live negotiated result) with the limiting
("least-capable") **listener** named — the listener whose `caps` floors the choice, computed
client-side by mirroring the [04 §4.3.2](../04-clock-and-groups.md) least-common-capable
rule over `caps.render === true` members (sink-less members never constrain it). Each field
is **auto by default / read-only**, with an explicit **override** toggle (D3/D4 are 🟡
defaults). Codec set is `"pcm"|"opus"` with **`pcm` the mandatory baseline** (always
selectable); FEC `"none"|"xorParity"|"duplicate"` (XOR default, D4). Overrides PATCH the
group; an infeasible override (below least-capable feasibility) returns `422 unprocessable`
(08 §E.4) surfaced in `ErrorBanner`.

**Transport (09 §5).** Radio **UDP unicast** (default) / **TCP fallback** (D2); a change
PATCHes the group profile/transport. (08 §0.7 GroupRecord has no separate `transport.state`;
`playing` is the play-state bool.)

**Play/Stop (09 §3, §5).** `playGroup`/`stopGroup` hit `POST /api/v1/groups/{id}/play|stop`
(08 §F.3/F.4), which flip the replicated `GroupRecord.playing` bool and **fan out to the
master** over mTLS. The button shows a spinner until `groupStatus.playing` (or the next WS
snapshot's `group.playing`) reflects the new state. If the master is a peer the call proxies
— show the `ProxyHint`; proxy failure (`502 proxy_failed`) → offline treatment + error toast.
Play is disabled when no media is selected (09 §3 "select media →" affordance) — the server
also guards with `409 conflict` "no media selected" (08 §F.3).

**Membership move (09 §5 flow 2).** Moving a node rewrites `memberNodeIds[]` on **both**
affected groups as a **single transactional config write** (08 §E.4, one ConfigDoc, LWW per
§6.5/§7). `moveNode` issues the PATCH; server-side re-election + profile re-negotiation
happen ([04 §4.6](../04-clock-and-groups.md)); the live profile updates from `/status`. The
synthetic **"Unassigned"** bucket is the UI's name for adopted nodes in their own solo group
not yet placed (09 §5; per glossary a node is always in exactly one group).

**Optimistic concurrency (09 §0, 08 §0.5).** Every mutating call sends
`If-Match: <configVersion>` (seeded from the live snapshot's `version`). On **409
version_conflict** the screen reloads the current doc (the WS push already refreshes it),
shows what changed, and asks the user to reapply via `ErrorBanner`'s **Reload & reapply** —
never silently overwrites. On **412** the client simply was missing the header (a bug guard).

**Shared screen states (09 §0):** `loading` → `Skeleton` (controls disabled, not hidden);
`empty` → "No groups yet — create one in **Groups**" (Dashboard) / big New-group prompt
(Groups), Unassigned still lists adopted nodes; `error` → per-card `ErrorBanner` (the rest
still render); `offline` → dimming + `OfflineChip`; `uninitialized` → never reached (P0.4
router gates it).

**No new A.12 tunables are introduced.** The only A.12 numbers this piece reads are display
constants: `rate = 48000`, `FramesPerChunk = 480`, `fecK = 8`, `interleave = 4` (A.12 Audio
canonical / FEC XOR) shown read-only in the profile sub-line, and the sync warn edge derived
from `HardErrSamp = 2400 samples (50 ms)` + the A.13 P4 sub-ms target. The ~1 Hz `/status`
poll cadence matches 09 §3 ("polled ~1 Hz"); it is a UI cadence, not an A.12 tunable.

## 6. Import constraints

This piece is **TypeScript/Svelte under `web/`** — the Go `internal/web` layering rule
(01 §2 rule 1: `web` must not import `group`/`stream/*`/`audio/*`) is satisfied **by
construction**: the browser app reaches the engine only over HTTP/WS, never via Go imports.
The frontend module-boundary discipline this piece must hold:

- Routes/components import only from `web/src/lib/*` and sibling `web/src/components/*` — they
  never embed `fetch` URLs inline; all REST goes through `groupActions.ts` (built on P0.4's
  `lib/api.ts`), all live data through the P0.4 `state` store + `groupStatus.ts`.
- `groups.ts`/`groupStatus.ts`/`groupActions.ts` depend only on P0.4's `lib/ws.ts`,
  `lib/api.ts`, `lib/format.ts` and stdlib `svelte/store` — no cross-imports into other
  screens' component trees.
- No dependency on any `media/web` `editor/*`, `playlist.ts`, `timecode.ts`, transfer, or
  screenshot code (excluded by the copy strategy).
- Endpoint paths are exactly the canonical `/api/v1/...` from 08 (no ad-hoc routes); 08 is
  authoritative on the wire (09 §0 dependency note).

## 9. Risks / open questions

1. **P0.4 / P2.7 specs do not yet exist in `doc/impl/`.** This plan assumes P0.4 owns the
   UI scaffold (router, auth shell, nav, re-homed `lib/ws.ts`/`api.ts`/`format.ts`, design
   tokens, shared state primitives) and P2.7 owns the live telemetry backend
   (`GET /api/v1/groups/{id}/status` + optional WS group projection). **Blocking if the
   split differs** — confirm: (a) does P0.4 provide shared `ErrorBanner`/`Skeleton`/
   `ProxyHint`/`OfflineChip` and a re-homed `lib/dnd.ts`? (this piece consumes them if so,
   else creates from the `ConfirmModal`/`MediaBin` patterns); (b) does P2.7 stream group
   telemetry over the WS hub, or is ~1 Hz polling of §G.2 the only source? `groupStatus.ts`
   is written to prefer WS and fall back to poll, so it works either way, but the WS shape
   (if any) must match P0.2's `StateSnapshot` extension.

2. **Sync-error warn threshold (⚠) is under-specified in 09/06.** 09 §3 says "over a
   threshold … per [06]" but 06 is not in this piece's read set and A.12 fixes only
   `HardErrSamp = 2400 samples (50 ms)` (the reseek bound) and the A.13 P4 "sub-ms" target.
   I set `SYNC_WARN_US = 1000` (1 ms — above the sub-ms target, well below the 50 ms reseek
   bound). **Confirm the warn edge with the 06/output-scheduling author** before final.

3. **`StateSnapshot.config` vs separate config GETs.** P0.2 §9 left open whether the seam
   returns a web-owned `ConfigView` projection (recommended there) or the literal
   `state.ConfigDoc`. This piece consumes the **`ConfigView` projection over WS** (the P0.2
   recommendation). If the orchestrator chose the literal `state.ConfigDoc`, the TS types in
   `groups.ts` change field names accordingly (non-blocking — same fields, possibly different
   casing). Also: 09 §3 lists the static structure as composed from `cluster/info`+`nodes`+
   `groups` GETs; this piece prefers the **single live WS snapshot** (reactive, no polling) —
   confirm that is acceptable vs. explicit per-screen GETs.

4. **Test runner.** Vitest + @testing-library/svelte are assumed wired by P0.4. If not, this
   piece adds them to `web/package.json` devDeps — flag so it is not duplicated.

5. **Drag-and-drop is progressive enhancement.** `MemberTransfer` ships multiselect+move as
   the baseline (always works, accessible); drag is layered on if a re-homed `lib/dnd.ts`
   exists. Non-blocking.
