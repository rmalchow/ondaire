## 1. Purpose & scope

P6.1 is a **verification-and-wiring** piece, not a new subsystem: it proves that *N
independent concurrent groups* each run their own `clock.Server`/`clock.Follower`,
`group.Engine` + sample-domain `Timeline`, stream origin, and election outcome with **zero
cross-talk** (one master's clock/timeline/`streamGen`/origin never leaks into another
group), and that **channel-role audio fan-out is correct** per [06 §5.1](../06-audio-output-scheduling.md#51-channel-role-stereo--left--right):
`stereo` passes through, `left` takes source ch0 and fans it to *all* output channels,
`right` takes source ch1 and fans it to all — so a physical L/R pair forms a stable stereo
image. It lands (a) the small amount of *missing* production code the verification needs —
the canonical channel-select helper extracted to a testable form, and the per-group engine-set
**registry** that `cmd/ensemble` uses to run many groups in one process — and (b) the
table-driven + concurrency tests that are the actual P6 deliverable (A.13 P6: "L/R stereo
pair forms a stable image; per-node delay trim works"; "groups, channel roles"). It owns the
multi-group lifecycle wiring and the channel/gain correctness proofs; it does **not** redefine
the engine (P3.2), the clock (P3.1), the election (P2.3), or the render loop (P4.9).

## 4. Public API (canonical names — README §6, 04 §4.7, 06 §5)

```go
package render // internal/audio/render (P4.9's package; P6.1 adds channel.go)

// Channel is this node's role within the canonical stereo pair (README §6.5, D13, 06 §5.1).
type Channel int

const (
    ChannelStereo Channel = iota // passthrough: src ch0->out ch0, src ch1->out ch1
    ChannelLeft                  // take src ch0, fan out to ALL output channels
    ChannelRight                 // take src ch1, fan out to ALL output channels
)

// ParseChannel maps the §6.5 JSON string enum to a Channel. "" => ChannelStereo.
// Unknown => error (caller defaults to stereo and logs).
func ParseChannel(s string) (Channel, error) // "stereo"|"left"|"right"

func (c Channel) String() string // back to the §6.5 string enum

// SelectChannel returns the source sample feeding output channel oc, per 06 §5.1.
// srcFrame is one interleaved canonical-stereo source frame (len == source channels,
// normally 2). outCh is the sink's output channel count; oc in [0,outCh). It is the
// per-frame pick the producer applies AFTER resampling and BEFORE gain (06 §5, §2.2).
func SelectChannel(srcFrame []float32, role Channel, outCh, oc int) float32

// GainLinear converts NodeRecord.GainDB to a linear amplitude scale (06 §5.2).
// 0 dB short-circuits to 1.0 (no multiply). Clamping to [-1,1] is the sink's job.
func GainLinear(db float64) float32
```

```go
package group // internal/group (P3.2's package; P6.1 adds registry.go)

// Registry runs one group.Engine per group the node knows about, fanning a single
// replicated-config + election update out to each independent per-group Engine.Apply.
// It is the multi-group concurrency surface: N groups, N Engines, N Timelines, N clock
// roles, N origins — fully independent (04 §4.1.2, §4.6 "exactly one group per node" for
// SELF, but the node may HOST/observe many group records; only self's group runs render).
type Registry struct { /* mu; self string; groups map[string]*groupRuntime; mk RuntimeFactory */ }

// RuntimeFactory builds the per-group Hooks + Timeline + ClockSource for a given group id,
// wired by cmd/ensemble to that group's clock.Server/Follower, stream origin/receiver, and
// (only for self's own group, when Render=true) the audio render loop. Keeps group from
// importing clock/stream/audio (Hooks seam, 01 §2 / A.14.3 discipline).
type RuntimeFactory func(groupID string) Hooks

func NewRegistry(selfID string, mk RuntimeFactory) *Registry

// OnState is called on every gossip/election event. doc is the replicated ConfigDoc;
// outcomes is the per-group election result (P2.3 cluster.GroupElections.Recompute output,
// keyed by groupID). It (a) creates a runtime for any new GroupRecord, (b) drops runtimes
// for removed groups (mirrors cluster.GroupElections.Drop), and (c) calls Engine.Apply with
// the per-group Inputs derived from doc+outcome — each group independently.
func (r *Registry) OnState(doc state.ConfigDoc, outcomes map[string]cluster.Outcome)

// Decisions returns the last applied Decision per group (read-only snapshot, for UI/status
// surfaced via cmd; web never imports group — A.14.3).
func (r *Registry) Decisions() map[string]Decision

// groupRuntime (unexported): one group's *Engine + its bound Hooks + last Inputs.
```

The engine state machine (`Engine`, `Hooks`, `Inputs`, `Decision`, `Recompute`, `Apply`), the
`Timeline`/`FollowerTimeline`/`MasterTimeline`, `ClockSource` adapters, and `NegotiateProfile`
are **P3.2's** — P6.1 does not redefine them; it instantiates one set **per group**.

## 5. Behavior & algorithms

**Per-group independence (04 §4.1.2, §4.6, A.5).** Each group is its own election domain and
its own clock domain. The `Registry` holds an independent `*Engine` per `GroupRecord`; on each
`OnState` it derives a *separate* `group.Inputs` for each group from `(doc.Groups[g],
outcomes[g])` and calls that group's `Engine.Apply` — so a master change, `streamGen` bump,
profile renegotiation, or failover in group A **never** touches group B's engine, timeline,
clock server, or origin. This mirrors 04 §4.1.2 ("Ensemble runs one clock server per group, on
that group's master; each follower pings *its* master only") and 04 §4.6's per-operation
"Re-run for both groups" semantics — the cross-group isolation is the spec invariant P6.1
exists to prove. Election independence is already provided by P2.3's `GroupElections`
(`map[groupID]*Election`); P6.1 verifies the *engine-side* independence end-to-end.

> Note on "exactly one group per node" (README §6.2): *self* is a member of exactly one group
> and renders only that one. A node may still **observe** (and, as master, **originate** for)
> other groups' records via the replicated ConfigDoc; the Registry models all known groups so
> the wiring is uniform, but only self's own group ever drives the local render loop (06 §1.5:
> a `Render=false` node never starts the render loop at all). The multi-group concurrency under
> test is therefore primarily exercised by a single process hosting several groups in tests and
> by a master originating for its group while the cluster runs others.

**Channel role fan-out (06 §5.1 — the correctness core).** The canonical stream is **stereo**.
`SelectChannel` implements the §5.1 table exactly, applied per frame in the producer *after*
resampling, *before* gain (06 §5, §2.2):

| `role` | mapping (for every output channel `oc`) |
|---|---|
| `ChannelStereo` | passthrough: `oc==0 ⇒ srcFrame[0]`, `oc==1 ⇒ srcFrame[1]`; if `oc ≥ len(srcFrame)` clamp to last available channel |
| `ChannelLeft` | always `srcFrame[0]` — source ch0 fanned to **all** output channels |
| `ChannelRight` | always `srcFrame[1]` — source ch1 fanned to **all** output channels (return `0` if the source has no ch1) |

A stereo *pair* is two physical nodes — one `ChannelLeft`, one `ChannelRight` — each fanning
its selected source channel to **both** of its own output pins, so a stereo DAC plays the same
mono on both pins (06 §5.1). This is what produces "a stable stereo image" (A.13 P6).

**Gain (06 §5.2).** `GainLinear(db)`: `g = 10^(db/20)`, with `0 dB ⇒ 1.0` short-circuit;
applied after channel-select (`out = g · SelectChannel(...)`). Clamping to `[-1,1]` is the
sink's job, not the gain stage's.

**`HWDelayUs` (06 §5.3) — verified, not re-implemented.** P4.9/P3.2 already apply the per-node
trim as an integer sample offset on the *want* side: `hwDelayOffset =
round(HWDelayUs · 1e-6 · Rate)`; `wantContent = sampleIndex + hwDelayOffset` (06 §3.2/§5.3,
A.4). P6.1's job (A.13 P6 "per-node delay trim works") is to assert this offset is honored end-
to-end: a positive `HWDelayUs` advances the want target (node plays *earlier* in content to
compensate a *later* acoustic arrival), held **exactly** in steady state by the drift loop.

**Quoted A.12 tunables (verbatim, single source of truth).** P6.1 introduces no new tunable;
the values it asserts against are all A.12's:

> Audio canonical — **rate / channels / FramesPerChunk = 48000 / 2 / 480 (10 ms)**
> Clock — **window N (wired·WiFi) / alpha (wired·WiFi) / ping = 8·16 / 0.15·0.10 / 1 s (500 ms
> initial lock)**; Clock RPC **timeout = 200 ms**
> Drift PI — **Kp / Ki / MaxPPM / IntegralClamp = 0.05 / 0.005 / 200 ppm / 200**; **tick /
> HardErrSamp = 20 ms / 2400 samples (50 ms)**
> Ports — **control mTLS / clock UDP / audio UDP = 8443 / 9000 / 9100** (per-group clock/audio
> sockets; on a single box extra nodes take +1 — README §4)

(The orphan enter/exit `MinDelay()` hysteresis thresholds remain **un-pinned** in A.12 — the
same open item P3.2 raised; P6.1 inherits it, see §9 R3.)

`internal/audio/render` (`channel.go`) — a leaf of the audio pipeline. MAY import: **stdlib
only** (`math` for `GainLinear`, `fmt`/`errors` for `ParseChannel`). It MUST NOT import

`internal/group` (`registry.go`) — MAY import: `internal/state` (`state.ConfigDoc`,
`GroupRecord`, `NodeRecord`), `internal/cluster` (`cluster.Outcome` — the election result type,
consumed as a plain input). It MUST NOT import `internal/stream/*`, `internal/audio/*`, or
`internal/web`: the per-group clock/origin/receiver/render lifecycle is reached **only** through
the P3.2 `Hooks` function-value seam supplied by the `RuntimeFactory` (wired in `cmd/ensemble`),
exactly as P3.2 mandates and mirroring the `web.Deps` discipline (A.14.3 / 01 §2). This keeps
the dependency graph acyclic and the registry unit-testable with fake `Hooks`.

### `internal/audio/render/channel_test.go`

| Case | Asserts |
|---|---|

### `internal/group/registry_test.go`

| Case | Asserts |
|---|---|
| add group A then group B | two distinct `*groupRuntime`s created; `Decisions()` has both; each `Engine.Apply` called with **its own** `Inputs` (group A's members ≠ group B's). |

### `internal/group/multigroup_test.go` (integration, still pure-Go)

Drives a `Registry` (or two, one per simulated node) with fake `Hooks` recording lifecycle
calls, a fake `ChunkMetaSource`, and the `MasterClock`/`OrphanClock` adapters, through a
scripted multi-group sequence:

- Two groups A (master n1, follower n2) and B (master n3, follower n4) run **concurrently**;
  assert each has its own master, its own `streamGen` sequence, and its own `Timeline` anchor —
  bumping A's `streamGen` (profile change / master change) does **not** change B's.
- **Failover isolation:** "kill" A's master (drop n1 from A's alive set) ⇒ A re-elects + bumps
  A's `streamGen` + A's engines re-point (T4/T5); assert B's election outcome, `streamGen`,

- **Timeline non-interference:** two `FollowerTimeline`s anchored on different

  clock (no shared mutable state) — verifies the per-group timeline isolation that 04 §4.1.2
  requires.
- **Channel role across the pair in a group:** group A's two listeners carry `Channel=left`
  and `Channel=right`; assert the producer (via `SelectChannel`) yields a coherent L/R image
  while both share group A's single `Timeline` (sample-aligned) — ties §5.1 correctness to the
  multi-group context.

Assertions trace to A.13 P6 and the 04 §4.6 lifecycle table (per-group "Re-run for both
groups" → here, "re-run isolates to the affected group").

> The wall-clock "24 h soak / sub-ms over 10 min" figures are P4/P7 system tests (real WiFi +
> analyzer), not P6.1 units. P6.1's automated bar is the *logical* independence + channel
> correctness above; the "audibly tightens / stable image" half is the manual A.13 P6 hardware

## 9. Risks / open questions

1. **R1 — channel-helper ownership overlap with P4.9 (coordination, low risk).** P4.9 may have
   already inlined `pickChannel`/`gainLinear` (copied from media). P6.1 promotes them to the
   exported `SelectChannel`/`GainLinear` in `channel.go` and rewires P4.9's call-sites. This is
   a *single-package* refactor with no behavior change, but the two specs must agree on who
   owns the file. **Proposed resolution:** `channel.go` (the helpers + `Channel` enum) is
   **P6.1's**; P4.9's `render` files call into it. Confirm P4.9 does not also define a
   conflicting `Channel`/`pickChannel`. **Needs a one-line coordination note in P4.9's spec.**

2. **R2 — "many groups in one process" vs "exactly one group per node" (clarification).**
   README §6.2 says a node is a member of exactly one group. The `Registry` nonetheless models
   *all* groups it observes (so a master can originate for its group while the wiring stays
   uniform, and so multi-group concurrency is exercisable in a single test process). This is
   consistent (self renders only its own group; 06 §1.5), but worth an explicit confirmation
   that the Registry-hosts-many model is the intended `cmd` shape rather than a one-group
   singleton. **Recommend: Registry (uniform), as specified.**

3. **R3 — orphan hysteresis thresholds still not in A.12 (inherited from P3.2).** The
   multi-group failover test wants to assert a follower in group A orphans on sync loss without
   flapping; the enter/exit `MinDelay()` thresholds + N-window count are **not pinned** in A.12
   (P3.2 §9 R1 raised the same). P6.1 tests the *logical* orphan transition via injected
   `MinDelayOK`, sidestepping the numeric gap, but the system-level "stable image" gate needs
   the real thresholds. **Blocking-ish for the hardware acceptance only:** orchestrator should
   ratify P3.2's proposed defaults (enter > 15 ms, exit < 8 ms, N = 3) into A.12.

4. **R4 — `cluster.Outcome` import in `group` (thinness).** `Registry.OnState` takes
   `map[string]cluster.Outcome`, adding a `group → cluster` import edge. P3.2 deliberately kept
   `group` thin by passing `MasterID`/`Generation` as scalars in `Inputs`. To preserve that,
   `OnState` could instead take a `map[string]group.ElectionResult` (a local 3-field struct)
   that cmd populates from `cluster.Outcome`, avoiding the import. **Open:** accept the thin
   `cluster.Outcome` import (simpler) or mirror it as a local struct (purer). **Recommend the
   local struct** to keep `group` free of `cluster` and fully fake-testable. Confirm with
   orchestrator; either is a one-type change.

5. **R5 — `ParseChannel("")` default.** §6.5 lists `Channel ∈ {"stereo","left","right"}`; the
   ConfigDoc default for an un-set field is empty string. P6.1 maps `""⇒ChannelStereo`
   (sensible default = full passthrough). If the schema (P2.1/07) instead guarantees `"stereo"`
   is always written explicitly, the `""` branch is dead-but-harmless. Non-blocking.
