## 1. Purpose & scope

Implement **screen 6, "Node detail"** ([09 §6](../09-ui-screens.md)) for *any* node in the
cluster (serving node or peer via proxy), and **finalize** the **Dashboard** (screen 3) and
**Settings** (screen 8) so the seven-screen UI is complete and consistent. Node detail
renders: **Identity** (rename + read-only id/group/master marker); a **conditional Audio-output
section** shown only when `caps.render === true` — **channel role** (stereo/left/right),
**gain** (dB), and **`HWDelayUs`** as **manual entry** (slider + numeric µs field) with a
**Calibration helper** card (Play test signal via `POST /api/v1/calibrate/play`, judge by
ear/phone-mic, type the trim, watch live sync error converge); the **"Control / media only"**
panel that replaces those controls when `caps.render === false` (variant B); an
**always-shown Capabilities & audio backends** panel that renders the effective structured
`Capabilities` (render flag, probed sinks tagged precise/coarse, encode/decode codecs, FEC,
max rate) with **per-node disable toggles** (mask a probed backend/codec/FEC) and a **force
control-only (`render:false`)** toggle that flips A↔B; a **Network** block (addrs + cert
fingerprint); and **Save / Revert** with `If-Match`. This is **pure presentation + action
wiring**: it owns no protocol or config-write logic of its own (those live behind P6.2's node
config / capability-mask / calibration endpoints). Finalization of Dashboard/Settings is
wiring the now-complete cross-screen links (member row → Node detail, sink-less tag →
Capabilities anchor) and closing the deferred items those sibling specs left open (§5.6).

export interface NodeDetailView {
  id: string
  name: string
  addrs: string[]                       // drive the allowlist (README §6.5)
  hwDelayUs: number                     // integer µs; "Hardware delay (HWDelayUs)" (09 §6 naming note)
  channel: 'stereo' | 'left' | 'right'  // D13
  gainDb: number
  caps: Capabilities                    // EFFECTIVE = detected(runtime) ∩ enabled(config) (D16)
  fingerprint?: string                  // node cert fingerprint (Network panel)
  certSignedByCa?: boolean              // header cert status
  online?: boolean                      // gossip/health (drives offline read-only mode)
  groupId?: string                      // current group (Identity + live-sync selector)
  isMaster?: boolean                    // elected master of its group (master marker)
}

// PATCH body for /api/v1/nodes/{id} (D.3). All fields optional (partial update).
// `capabilities` is the per-node MASK that re-shapes effective caps (07; D16) — the
// caller sends the *desired enabled set*; the node re-probes/re-masks and re-advertises.
export interface NodePatch {
  name?: string
  channel?: 'stereo' | 'left' | 'right'
  gainDb?: number
  hwDelayUs?: number
  capabilities?: CapabilityMask
}
// Per-node enable/disable mask (subtracts probed paths; forces render:false). Shape mirrors
// the effective Capabilities so the server intersects probed ∩ this. (08 §D.3 "capabilities?";
// 07 per-node config.)
export interface CapabilityMask {
  render?: boolean        // false => force control-only (sink-less)
  sinks?: string[]        // enabled backend names (subset of probed)
  encode?: string[]
  decode?: string[]
  fec?: string[]
}

// ---- Reads ----
export function getNode(id: string): Promise<{ version: number; node: NodeDetailView }>   // GET /api/v1/nodes/{id} (D.2)

// ---- Writes (require If-Match: <version>; threaded by the scaffold apiClient) ----
export function patchNode(id: string, patch: NodePatch, ifMatch: number):
  Promise<{ version: number; node: NodeDetailView }>                                       // PATCH /api/v1/nodes/{id} (D.3)

// ---- Calibration (transient; NO If-Match — does not write the ConfigDoc) ----
export interface CalibratePlayReq { groupId?: string; nodeIds?: string[]; durationSec: number } // exactly one of groupId/nodeIds
export interface CalibratePlayResp { playedOn: string[]; durationSec: number; warnings: string[] }
export function calibratePlay(req: CalibratePlayReq): Promise<CalibratePlayResp>            // POST /api/v1/calibrate/play (F2.1)

## 5. Behavior & algorithms (cite docs; A.12/A.10b numbers verbatim)

### 5.1 Data sources & composition (09 §6 "Data sources")
- Node record incl. effective structured `Capabilities`: `GET /api/v1/nodes/{id}` (08 §D.2)
  — **read-only, local** (the record lives in the replicated doc, any node answers). The
  screen prefers the **live `state` snapshot** projection if the scaffold pushes one (reactive,
  no polling) and falls back to a one-shot `getNode(id)` on mount; either way it seeds
  `If-Match` from the snapshot/`ETag` `version`.
- Live sync error: `GET /api/v1/groups/{id}/status` (08 §G.2) for the node's **current group**,
  consumed via **P3.3's `groupStatus` store** (`pollGroupStatus(node.groupId)` while the
  Calibration helper is visible). `liveSyncErrorUs` selects this node's `members[].syncErrorUs`;
  the master shows `—` (it is the reference) and an offline/no-group node shows `null`.
- `Capabilities` is `detected(runtime) ∩ enabled(config)` (README §6.5 / D16): the node
  **probes available backends at runtime (D12), masks with per-node config, re-advertises** —
  this screen never re-derives the effective set itself; it renders what the record carries and
  only *previews* what a mask edit would do (`caps.previewRender`).

### 5.2 Audio-output section is conditional on `caps.render` (09 §6, D17, 06 §1.5)
- **`renderEnabled === true` (variant A):** show `ChannelRoleRadio` (stereo/left/right, D13 /
  06 §5.1), `GainSlider` (`GainDB`, 06 §5.2), and `HwDelayControl` for **`HWDelayUs`** as
  **manual entry** (06 §5.3) plus the `CalibrationHelper`.
- **`renderEnabled === false` (variant B):** **hide all three** (channel/gain/HWDelay) +
  helper; render `ControlMediaOnlyPanel` ("Control / media only — no audio output") explaining
  the node can still be **master/origin, media store, clock, and UI** but is **not a listener**
  (06 §1.5, 09 §6 variant B), and point at the Capabilities panel to re-enable a sink.
- `renderEnabled` is taken from the **draft** (`draft.capabilities.render ?? loaded.caps.render`),
  so toggling **force control-only** flips A↔B **live in the form** before save. Channel/gain/
  HWDelay edits sent for a node that ends up `render:false` are **ignored server-side** (no
  sink, 09 §6 "Data sources") — the UI simply stops showing them.

### 5.3 `HWDelayUs` field & the calibration helper (06 §5.3, A.10b, 09 §6 flow 2)
- **Field naming:** label it **"Hardware delay (HWDelayUs)"** and edit the `NodeRecord` field
  **`hwDelayUs`** (09 §6 spine-terminology note: use the field name `HWDelayUs`, not the
  `DelayUs` prose alias). Integer **µs**; two-way bound slider (0..20000 µs per the [09 §6
  wireframe](../09-ui-screens.md)) ↔ numeric box.
- **Calibration is MVP-manual (D21).** The helper steps mirror the [09 §6 wireframe](../09-ui-screens.md):
  (1) add the node to a group with a reference; (2) **Play test signal** → `POST
  /api/v1/calibrate/play {groupId | nodeIds, durationSec}` (08 §F2.1) — plays the **built-in
  click+tone signal synchronously** on the selected node(s); (3) **judge the offset by ear /
  phone-mic**; (4) **type the trim into `HWDelayUs` and Save** (`PATCH /api/v1/nodes/{id}`,
  08 §D.3). The helper reads the **live sync error** (`liveSyncErrorUs`) and shows ✔ when it
  trends sub-ms.
- **Built-in signal (A.10b, verbatim).** "Period = **1 s**, repeated: `[0, ~1 ms)` full-scale
  click; `[~1 ms, ~201 ms)` 1 kHz sine tone (~200 ms); `[~201 ms, 1 s)` silence; identical
  across nodes." Generated **in-process at the canonical rate** (no external file). The UI does
  not generate it — it only triggers playback and explains it.
- **Automated cross-correlation `/calibrate/measure` is a documented later enhancement
  (08 §F2.2, 06 §5.3, D21) — NOT in the MVP.** The helper must say so; no measure/upload UI.
- **Sign convention (06 §5.3, informational copy):** a **positive** `HWDelayUs` advances this
  node's want target (it plays *earlier* in content to compensate a *later* acoustic arrival),
  held **exactly** in steady state by the drift loop. The helper may surface this as a one-line
  hint; it must **not** invent a different sign.

### 5.4 Capabilities & audio backends panel (always shown — 09 §6, README §6.5, D16/D12/D17)
Renders the effective structured `Capabilities` object (**not** a flat string array):
- **`render`** flag — "yes (listener-capable)" / "no (control / media only)".
- **Sinks** — usable+enabled backends from `caps.sinks`, each tagged **precise** or **coarse**
  via `caps.sinkTier`: `alsa` (direct-ioctl) and `pipewire` → **precise**; `exec:aplay` /
  `exec:pw-play` → **coarse** (`Delay()` `ok=false`) per [06 §1.1/§1.3](../06-audio-output-scheduling.md).
- **Encode codecs** (`caps.encode`, what it can ORIGINATE as master) and **Decode codecs**
  (`caps.decode`, what it can PLAY as listener — shown **"—"** when not a listener), each a
  string-enum set (`"pcm"|"opus"`, `pcm` the mandatory baseline R2); **FEC** (`caps.fec`,
  `"none"|"xorParity"|"duplicate"`); **Max rate** (`caps.maxRate`).
- **Per-node disable toggles** — for each **technically-available (probed) path**, an
  enable/disable control (`CapabilityToggleRow`) that **masks** the runtime-detected capability
  via per-node config (07): disable a backend / codec / FEC. A path that was **never probed** is
  **not offered** (you cannot enable what the runtime did not discover, D12) — render it disabled
  with a "not available on this node" note. The mask is sent as `capabilities` in the `PATCH`
  (08 §D.3); the node re-probes/re-masks and re-advertises a narrower effective `Caps`; group
  profile re-negotiates if a now-removed codec/FEC was in use ([04], 09 §6 flow 3).
- **Force control-only (`render:false`)** — `ForceControlOnlyToggle` sets
  `draft.capabilities.render = false`, turning the node sink-less (variant B). Clearing it (with
  a probed sink available) restores rendering. The toggle's effect on A/B uses
  `caps.previewRender` (render = at least one enabled sink, 06 §1.5) so the form preview matches
  what the server will derive.

### 5.5 States, proxy, and optimistic concurrency (09 §6 states, §0; 08 §0.5/§0.6)
- **loading** → `Skeleton` (controls disabled, not hidden). **error** → not-found (node forgotten
  mid-edit) shows the envelope `code`/`message` + back to Cluster; save **409 version_conflict**
  → reload the doc + **Reload & reapply** (`Banner`), never silently overwrite. **uninitialized**
  → gated by the router.
- **sink-less (`render === false`)** is a **content variant, not an error** (variant B) — see §5.2.
- **offline** (`loaded.online === false`, or a proxied call times out / TLS-fails): **read-only**
  view from the gossiped ConfigDoc with an `OfflineChip` banner; **HWDelayUs / channel / gain,
  the calibration helper, and the capability toggles are disabled** (they need the live node to
  re-probe), last-known effective `Caps` shown dimmed. **Rename** (config-only) **may** still be
  allowed and labeled "applies when node returns" (09 §6 offline state). The `disabled` prop
  threaded into the audio + capability components is `offline || saving`.
- **Proxy (09 §0, 08 §0.6/§D.3):** `PATCH /api/v1/nodes/{id}` is **proxied to the owning node**
  (channel/HWDelayUs/gain take local effect on its renderer; capability edits change what it
  probes/masks); `POST /api/v1/calibrate/play` is **fan-out**. Show the `ProxyHint`

## 9. Risks / open questions

1. **(Blocking) Dependency-ID reconciliation: "P4.10 / P6.2" vs the existing FE spec set.**
   The orchestrator lists upstream **P4.10, P6.2**, but the impl docs already present in
   `doc/impl/` use a *different* numbering for the same symbols: the **UI scaffold** (router,
   `apiClient`, `If-Match` plumbing, `ConfirmModal`/`Banner`/`Field`/`CopyField`, `AppShell`,
   **Settings shell**) is owned by **P1.4**; **Cluster + `ControlMediaOnlyTag`/`ProxyHint` +
   `lib/cluster.ts`** by **P2.7**; **Dashboard + `lib/groups.ts`/`groupStatus.ts`/`syncfmt.ts` +
   the shared badges** by **P3.3**; the **sink/caps probe + AudioSink** by **P4.6/P2.6**; and the
   **node-config/calibration API** by the node API piece. **The orchestrator must confirm the
   mapping** — concretely: which piece exports (a) the `apiClient`/`router`/shared components this
   screen builds on, (b) `lib/groups.ts`+`groupStatus`+`syncfmt`, (c) the `GET/PATCH
   /api/v1/nodes/{id}` and `POST /api/v1/calibrate/play` handlers + view types. If those exports
   live under P4.10/P6.2 rather than P1.4/P2.7/P3.3, only the import paths/dep-IDs change; the
   component/store design here is unaffected.

2. **Per-node capability-mask wire shape (08 §D.3 `capabilities?`).** 08 §D.3 lists
   `capabilities?` in the PATCH body but does not fully specify whether the client sends the
   **desired enabled set** (intersected server-side with the probe) or a separate
   *disabled-list*. 07 (per-node config) is the authority. This spec models it as the **enabled
   set mask** (`CapabilityMask` mirroring `Capabilities`), matching D16's `detected ∩ enabled`. If
   P6.2/07 defines an explicit disable-list shape instead, only `lib/node.ts`'s PATCH body and
   `nodeStore.toggleCapability` change — confirm with the P6.2 / 07 author.

3. **Live node record: WS snapshot vs `getNode` poll.** 09 §6 lists the source as
   `GET /api/v1/nodes/{id}`, but the FE set generally prefers the single live `state` snapshot
   (reactive). This piece prefers the snapshot and falls back to `getNode` on mount. **Confirm the
   scaffold's `state` snapshot carries the full per-node `caps`** (not a trimmed projection); if
   trimmed, the screen must `getNode` on open and after each save.

4. **Sync-error warn edge `SYNC_WARN_US` (carried over from P3.3 §9).** With [06] now readable,
   A.12 fixes only `HardErrSamp = 2400 samples (50 ms)` (reseek bound) and A.13 P4 the "sub-ms"
   target; there is no explicit warn threshold. This piece keeps **`SYNC_WARN_US = 1000` µs**
   (1 ms) for the helper ✔/⚠ and the Dashboard, citing A.12 + A.13 P4. **Confirm the warn edge
   with the 06/output-scheduling author** before final (non-blocking; one constant).

5. **`pipewire` as a precise tier.** 09 §6's wireframe tags `pipewire` "precise", but [06 §1.1]
   enumerates only `alsa` (precise, direct ioctl) and `exec:aplay`/`exec:pw-play` (coarse) as
   compiled backends — there is no standalone `pipewire` backend in 06's registry. `caps.sinkTier`
   follows the **record's own precise/coarse signal where available** (the backend's
   `Backend.Precise`); absent a per-sink precise flag in the `Capabilities` JSON, it classifies by
   name (`alsa`→precise, `exec:*`→coarse) and treats any literal `"pipewire"` as precise per the
   wireframe. **Confirm whether the `caps.sinks` JSON carries a per-sink precise flag**; if so, use
   it directly instead of name-classification.

6. **Route param + nav slot for `/nodes/:id`.** Node detail is reached **contextually** (not a flat
   nav item, 09 §0/index). The P1.4 router must support a **path param** (`/nodes/:id`) and a
   back-link to the origin (Dashboard/Cluster/Groups). **Confirm P1.4's router exposes param
   routes + a `back()`/origin mechanism**; if it is hash-only without params, `NodeDetail` reads the
   id from the hash query and the deep-links adapt (non-blocking — same screen).

7. **Calibration target selection.** `POST /api/v1/calibrate/play` takes `groupId` **or**
   `nodeIds`. The helper (09 §6 flow 2) wants the operator to compare against a reference, so it
   defaults to **the node's current `groupId`** (plays on all members synchronously) with a
   fallback to **just this node** when it is solo/ungrouped. Confirm this matches the intended
   operator workflow (non-blocking; UI default only).
